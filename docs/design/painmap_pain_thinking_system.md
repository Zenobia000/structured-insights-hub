## §1 設計原則

### 1.1 為什麼要強制拆兩階段（v2 的最大變革）

v1 的 7 步流程把「痛點發想」與「商業驗證」混為一個 sprint，導致：

- **小白受眾被嚇跑**：第一次發想就要架收款連結、設計手作交付
- **AI 引導被破壞**：使用者一進入流程就在想 App / 收費 / 商業模式，正是 Sunny 講稿明令禁止的反模式
- **判斷力沒被訓練**：「真假痛點」這個獨立判斷被「能不能賺錢」蓋過

v2 強制分階段，原因如下：

| 階段 | 主問題 | 主方法 | 主產出 |
| :--- | :--- | :--- | :--- |
| **階段一** | 這是真痛點還是假痛點？ | AI 證據蒐集 + 真人訪談規劃 + 自我判斷 | 書面「真/假」判斷 |
| **階段二** | 能不能用這個痛點建立生意？ | 手作交付 + 預售 + 72 小時 sprint | 第一筆真實付款 |

兩個階段**不可逆向**：
- 階段一沒過 → 階段二沒意義（驗證假題目）
- 階段一過 → 階段二**未必過**（真痛點未必是好生意）

### 1.2 為什麼把 TRIZ/BCG/SECI 重組為「鏡頭+引擎」

並列三個框架時，操作者必須自己決定「先用哪個看」，這是定義漂移的溫床。v2 維持 v1 的鏡頭管線設計，但**重新分配**：

| 理論 | 階段歸屬 | 角色 |
| :--- | :--- | :--- |
| **SECI** | 階段一 | 鏡頭——從口語抱怨到證據組合到真人確認 |
| **TRIZ** | 階段一 | 鏡頭——找出抱怨背後的矛盾本質 |
| **BCG DRI** | **階段二** | 鏡頭——驗證後決定攻擊層次（**不在發想階段使用**） |
| **第一性原理** | 跨階段 | 引擎——還原成物理量；#1、#2 在階段一，#3、#4 在階段二 |
| **蘇格拉底** | 跨階段 | 引擎——S1 定義 / S2 壓力測試貫穿全程 |

### 1.3 BCG DRI 為什麼不在階段一

DRI 回答「**怎麼解**」——這必須在「痛點是真的」確認後才有意義。在階段一就跳到 DRI，會把使用者拖進「想 App / 想商業模式」的反模式。

**唯一例外：** 階段一 Step 9 的真假判斷後，可以**用 1 句話標注**「如果是真的，初判可能屬於 D/R/I 哪一層」作為階段二的入口提示——但**不參與真假判斷**。

### 1.4 為什麼禁止跳步、禁止單鏡頭過度使用

**跳步反模式：** 聽到抱怨直接想「這要 Reshape」「這要做 App」，跳過 SECI 外化與 TRIZ 矛盾識別 → 把 AI 該怎麼用當成痛點。

**單鏡頭過度反模式：** 愛上 TRIZ 矛盾，把每個痛點硬塞進六矛盾之一 → 分類學壓垮現實。

引擎擋這兩個反模式：S1 強迫先定義；S2 強迫檢查定義漂移。

## §2 雙引擎（不重新定義，只組裝）

### 2.1 Elon 第一性原理 = 還原器

**完整定義見：** `first_principles_playbook.md §2`

本流程的物理量**分階段使用**：

| 物理量 | 一句話 | 階段 | 檢核時機 |
| :--- | :--- | :--- | :--- |
| **#1 具體的人** | 你能當下打電話給 3 位 | 階段一 | Step 2（5 要素檢核） |
| **#2 正在投入時間/成本的痛** | 替代方案有名字 | 階段一 | Step 2（5 要素檢核） |
| **#3 手工可交付的動作** | 今晚就能做一次給 1 個人 | **階段二** | sprint manual Hour 12 |
| **#4 付款通道** | 現在就有可分享的收款連結 | **階段二** | sprint manual Hour 24 |

**v2 重要修正：** v1 把 #2 寫成「正在**付錢**的痛」，會誤導發想階段就找付費客戶。v2 改為「正在**投入時間/成本**的痛」——只要 workaround 有名字、有時間/情緒/金錢成本投入，就成立。

### 2.2 蘇格拉底 = 詰問器

**完整定義見：** `socratic_first_principles_rnd_workflow.md §2, §3`

本流程使用 S1/S2 兩個工序，**每個鏡頭前後各跑一次**：

- **S1（鏡頭開始前）**：誰／何時／如何觀察到——三要素必齊。
- **S2（鏡頭結束前）**：
  1. **可證偽性**：什麼證據會讓我放棄這個痛點？
  2. **假因排除**：是不是其實是另一個痛點？
  3. **定義漂移**：起初定義 vs 現在定義是否同一件事？

## §3 三鏡頭詳解

### 3.1 鏡頭 1：SECI（橫跨整個階段一）

SECI 不再是一次性鏡頭，而是**三段使用**：

| SECI 工序 | 階段一位置 | 動作 |
| :--- | :--- | :--- |
| Socialization | Step 1 + Step 7 | 入口聽抱怨；訪談規劃時的真人現場確認 |
| Externalization | Step 2 | 抱怨變痛點 5 要素 |
| Combination | Step 5 + Step 6 | AI 蒐集公開證據；自己先猜+對照 AI 整理 |
| Internalization | **階段二** | 透過 sprint 把驗證結果變成下一輪的判斷力 |

**誤用警告：**
- ❌ 用 AI 生成合成 persona 取代 Socialization
- ❌ 在沒有具體場景時就跳到 Externalization

### 3.2 鏡頭 2：TRIZ（階段一 Step 3）

把痛點翻譯成「兩件事不能同時成立」的矛盾語言。

**六矛盾**（來自 workshop 講稿）：
1. 速度 vs 品質
2. 個人化 vs 規模化
3. 速度 vs 準確
4. 專家能力 vs 新手可用
5. 自動化 vs 人類判斷
6. 高頻實驗 vs 低風險

**輸出：** 矛盾標籤（**單選**） + 兩端具體描述。

**誤用警告：**
- ❌ 把矛盾標籤當痛點本身
- ❌ 為硬套 TRIZ 強塞矛盾 → 寫不出來退回 Step 2

### 3.3 鏡頭 3：BCG DRI（**階段二** — 不在階段一使用）

**理論：** Deploy（補丁）/ Reshape（流程）/ Invent（品類）三層 AI 應用。

**為什麼在階段二：** 這是「**怎麼解**」的判斷，必須在痛點被確認真實之後才有意義。

**階段一 Step 9 的角色：** 只在「真痛點」判定後，**附 1 句**初判 D/R/I 層次作為階段二入口，**不參與真假判斷**。

**階段二完整應用：** 見 `painmap_disruption_framework.md`。

## §4 階段一：痛點發想 9 步流程（不談錢）

> **使用方式：** 每一步都有**輸入 / 動作 / S1 詰問 / 物理量檢核 / 輸出 / 退出條件**。退出條件不滿足不能往下走。

### Step 1. 聆聽（SECI / Socialization 入口）

| 欄位 | 內容 |
| :--- | :--- |
| **輸入** | 一句口語抱怨（聽到的、不是解釋的） |
| **動作** | 把原句逐字寫下，不改寫；附上時間、地點、來源 |
| **S1 詰問** | （無） |
| **物理量檢核** | （無） |
| **輸出** | 原句逐字稿 + 1 個具體場景證據 |
| **退出條件** | 原句中可指出**至少一個具體可命名的人** |

### Step 2. 抱怨變痛點 5 要素（SECI / Externalization + S1 + 物理量 #1、#2）

| 欄位 | 內容 |
| :--- | :--- |
| **輸入** | Step 1 逐字稿 |
| **動作** | 改寫成「我每次要 ___，都會卡在 ___」格式；填 5 要素表 |
| **S1 詰問** | 痛點主人是誰／何時／如何觀察到？三要素齊嗎？ |
| **物理量檢核** | **#1**：3 人名單（有名字、聯絡方式、背景）<br>**#2**：他現在用什麼 workaround？這個 workaround 有具體名字嗎？ |
| **5 要素表** | 具體人群／具體場景／發生頻率／明確成本／現有解法不滿 |
| **輸出** | 痛點句 + 5 要素表 + 3 人名單 + workaround 名稱 |
| **退出條件** | 5 要素都具體（無「應該」「大概」）；3 人名單成立；workaround 有名字 |

### Step 3. 找矛盾（TRIZ）

| 欄位 | 內容 |
| :--- | :--- |
| **輸入** | Step 2 痛點句 + 5 要素 |
| **動作** | 從 6 矛盾中**單選一個**標籤；寫兩端具體描述 |
| **S1 詰問** | 「兩端各自具體指什麼？」（不能用抽象詞） |
| **物理量檢核** | （已在 Step 2 完成 #1、#2） |
| **輸出** | 矛盾標籤 + 兩端具體描述 |
| **退出條件** | 矛盾單選成立。複選或寫不出 → 退回 Step 2 |

### Step 4. 選 AI 工具角色

| 欄位 | 內容 |
| :--- | :--- |
| **輸入** | Step 3 矛盾 + Step 2 5 要素 |
| **動作** | 從工具地圖選 1 個 AI（ChatGPT Deep Research / Claude / Perplexity / Gemini） |
| **判準** | 第一次發想 → ChatGPT Deep Research；已有大量文字 → Claude；要查最新趨勢 → Perplexity / Gemini |
| **輸出** | 選定的 AI 工具 |
| **退出條件** | 知道你為什麼選這個工具（1 句話） |

### Step 5. 用 AI 找證據（SECI / Combination 加速）

| 欄位 | 內容 |
| :--- | :--- |
| **輸入** | Step 4 選定工具 + Step 1–3 全部產出 |
| **動作** | 複製 §7 的 8 題證據蒐集 prompt，**明確禁止 AI 設計產品/商業模式** |
| **S1 詰問** | （無——這一步是執行） |
| **物理量檢核** | （無） |
| **輸出** | AI 整理的 8 題回覆 |
| **退出條件** | AI 沒進入解決方案；8 題都有答 |

### Step 6. 自己先猜，再讀 AI 回覆（蘇格拉底自我論述前置）

| 欄位 | 內容 |
| :--- | :--- |
| **輸入** | Step 5 AI 回覆（**先不看**） |
| **動作** | 先寫 3 個猜測（誰最痛／哪場景最痛／可能假痛點）；再讀 AI 回覆並對照差異 |
| **S1 詰問** | 「我猜 vs AI 答的差異在哪？AI 補了什麼？AI 漏了什麼？」 |
| **物理量檢核** | （無） |
| **第二輪 prompt** | 請 AI 整理成「痛點判斷表」，挑出最值得優先研究的 1 個人群 |
| **輸出** | 猜測表 + 對照表 + 痛點判斷表 |
| **退出條件** | 4 個檢查點全通過；能寫出 3 個差異點 |

**為什麼要先猜：** 直接看 AI 回覆會被它牽著走（「啊它說的對！」是順著問題寫的錯覺）。先猜後對照才能訓練判斷力。

### Step 7. 規劃真人訪談（SECI / Socialization 現場確認）

| 欄位 | 內容 |
| :--- | :--- |
| **輸入** | Step 6 痛點判斷表 |
| **動作** | 從 AI 給的 5 種訪談對象挑 2 種；寫 3 個訪談題；列訪談禁則 |
| **S1 詰問** | 「我為什麼選這 2 種而非其他 3 種？」 |
| **物理量檢核** | 回頭看 #1：訪談對象是不是 Step 2 3 人名單裡的人？ |
| **訪談禁則** | 不推銷解法、不問「你會付多少」、只問現況 |
| **輸出** | 訪談對象清單 + 3 訪談題 + 訪談禁則確認 |
| **退出條件** | 至少 1 位有具體聯絡管道；3 訪談題不是推銷題 |

### Step 8. Pain Quality Score（教學評量工具）

| 欄位 | 內容 |
| :--- | :--- |
| **輸入** | Step 1–7 全部產出 |
| **動作** | 5 維度各 1–5 分，總分 25 |
| **5 維度** | 人群具體度／發生頻率／痛苦強度／現有解法不滿／證據可信度 |
| **分數帶** | 20–25 排訪談；15–19 縮人群；14 以下換題（**僅參考，不是決策**） |
| **輸出** | 5 維度分數 + 總分 |
| **退出條件** | 5 維度都打分；理解「分數是鏡子不是答案」 |

### Step 9. 真/假痛點判斷（階段一終點）

| 欄位 | 內容 |
| :--- | :--- |
| **輸入** | Step 1–8 全部產出 |
| **動作** | 書面回答「真痛點 / 假痛點 / 待訪談」+ 100 字理由 + 下一步 |
| **S1 詰問** | （已在 Step 6 完成自我論述） |
| **S2 壓力測試** | 1. 什麼證據會讓我放棄？<br>2. 是不是其實是另一個痛點？<br>3. Step 1 vs Step 7 是同一人同一情境嗎？ |
| **可選**（不參與真假判斷） | 1 句話初判 D/R/I 哪一層作為階段二入口 |
| **輸出** | 真假判斷卡（書面） |
| **退出條件** | 三個 S2 都有書面答案；判斷有 100 字理由；下一步明確 |

**階段一終點的 3 種出口：**

```
□ 真痛點   → 進入階段二（Step 7 訪談 → sprint manual）
□ 假痛點   → 退回 Step 1，換題目重來
□ 待訪談   → 留在 Step 7，先完成訪談再回 Step 9
```

## §5 雙模式輸出

### 5.1 模式切換規則

```
教學情境 → Pain Quality Score（25 分鏡子）
R&D 產出情境 → Validation Status（4 階標籤）
切換判準：受眾是「正在學的人」還是「正在做的人」
```

### 5.2 教學模式：Pain Quality Score（25 分鏡子）

**唯一用途：** 讓初學者看見「痛點被診斷的維度」。

**5 維度 ↔ 9 步對應：**

| # | 維度 | 1–5 分判準 | 對應步驟 |
| :-- | :--- | :--- | :--- |
| 1 | 人群具體度 | 1=抽象；5=能列 3 名具體人 | Step 2（物理量 #1） |
| 2 | 發生頻率 | 1=「應該會」；5=「最近一週 ≥3 次」 | Step 2（5 要素） |
| 3 | 痛苦強度 | 1=有點煩；5=矛盾兩端都被犧牲 | Step 2（成本）+ Step 3（TRIZ） |
| 4 | 現有解法不滿 | 1=說不出；5=能指名 + ≥3 不滿點 | Step 2（物理量 #2） |
| 5 | 證據可信度 | 1=我猜；5=多來源證據 + 可訪談 | Step 5–7 |

**寫死警示：**
> 「Pain Quality Score 是診斷的鏡子，不是診斷的結果。
> 24 分仍可能假；14 分仍可能真。
> 真假判斷的真正動作是 Step 7 真人訪談。」

### 5.3 產出模式：Validation Status（4 階標籤）

引用 `painmap_prd.md §3 Epic 2`：

| 狀態 | 進入條件 | 階段 |
| :--- | :--- | :--- |
| `draft` | Step 1 完成 | 階段一 |
| `structured` | Step 1–9 全部通過、判定真痛點 | 階段一 |
| `verified_interview` | 階段二 sprint Hour 24–48：≥ 5 位訪談 + 紅隊報告 | 階段二 |
| `verified_payment` | 階段二 sprint Hour 48–72：≥ 1 位真實付款 | 階段二 |

**鐵律（引自 `painmap_moat_design.md M3`）：** 產出模式中**禁止輸出 0–25 分數**。違反 = 違反 painmap 反焦慮品牌定位。

### 5.4 模式切換的書面交付

從教學模式過渡到產出模式時，附轉換卡：


## §8 反模式（Anti-Patterns）

### 8.1 跳階段反模式（v2 新增）

**症狀：** 痛點還沒判定真假，就跳到「設計手作交付」「架收款連結」「跑 72 小時 sprint」。

**根因：** 把階段一與階段二混為一個流程。

**解方：** 階段一 Step 9 退出條件鎖死「書面真假判斷」；沒判定真痛點不能進階段二。

### 8.2 跳步反模式

**症狀：** 跳過 Step 6（先猜後對照），直接讀 AI 回覆並接受。

**根因：** 把 AI 當答案而非證據。

**解方：** Step 6 強制要求書面猜測；沒寫猜測不能讀 AI 回覆。

### 8.3 單鏡頭過度反模式

**症狀：** 把每個痛點硬塞進 6 矛盾之一。

**根因：** 把分類學當現實。

**解方：** Step 9 S2 的「假因排除」要寫至少 1 個替代假設並排除。

### 8.4 把分數當決策反模式（IdeaCheck 病）

**症狀：** Pain Quality Score 22 分就跳到「做 App」。

**根因：** 把鏡子（教學工具）當結果（產出標籤）。

**解方：** §5.4 強制轉換卡——任何進入 R&D 的痛點都要附 Validation Status。22 分 + 沒訪談 = `draft`。

### 8.5 取悅 AI 反模式

**症狀：** 問 AI「這痛點 OK 嗎」，AI 給正向回饋後就行動。

**根因：** AI 是類比機器，會基於相似成功案例給正向回饋。

**解方：** Step 9 三個 S2 詰問**禁止用 AI 直接生成答案**——你必須自己寫。

### 8.6 合成 persona 反模式

**症狀：** 用 AI 生成虛構受訪者跳過真人訪談。

**根因：** Step 1 沒有真人來源。

**解方：** Step 1 退出條件鎖死「至少一個具體可命名的人」；Step 7 訪談對象必須在 Step 2 3 人名單內。

## §9 附錄 — 補習班老師案例完整走完階段一 9 步

### Step 1. 聆聽

**逐字原句**（2026-04-15 與林老師訪談）：
> 「我每週六晚上要寫 30 個學生的家長回報，平常週間都要記筆記但常漏，到週末翻 7 次小考成績單、翻群組對話、翻學生作業，常寫到半夜兩點。寫完還要怕語氣太冷或太重，傷家長感情。」

### Step 2. 5 要素 + 物理量 #1、#2

**痛點句：** 「我每週六要寫 30 則家長回報，都會卡在資料散在週間 7 次小考、要寫得具體、不能傷感情。」

**5 要素：**
- 具體人群：補習班國中部老師（30–60 學生規模）
- 具體場景：每週六晚上 21:00–02:30
- 發生頻率：每週 1 次（每年 50+ 次）
- 明確成本：5–6 小時/週 + 半夜情緒成本
- 現有解法不滿：LINE+Excel 拼湊（找資料慢）；Notion 模板（貼來貼去更慢）；ChatGPT 寫稿（罐頭感）

**物理量 #1（3 人名單）：** 林老師 / 王老師 / 陳老師（皆有名+聯絡方式）
**物理量 #2（workaround 名稱）：** LINE + Excel 成績表 + 翻群組對話

### Step 3. 找矛盾（TRIZ）

**矛盾標籤：** **#2 個人化 vs 規模化**
- A 端（個人化）：家長要看見「我的孩子」被個別關照
- B 端（規模化）：老師一週只有 2–3 小時可寫 30 則
- 通常犧牲：A 端（罐頭訊息）

### Step 4. 選 AI 工具

**選定：** ChatGPT Deep Research（第一次發想，需要外部證據）

### Step 5. 用 AI 找證據

執行 §7 prompt，AI 回覆 8 題（略，含 5 個工具評論、3 個論壇貼文、2 篇補教產業文章證據）。

### Step 6. 自己先猜+讀 AI

**我的猜測：**
- 最痛的人：補習班導師（負責全班）
- 最常場景：段考後一週
- 可能假痛點：也許大型補習班用班務系統已解決

**AI 補了什麼：** 「行政老師」也很痛（之前沒想到——他們要彙總所有導師的回報）
**AI 漏了什麼：** 沒提到「家長很急 vs 老師寫不完」的緊張感（我從訪談聽到的）
**痛點判斷表：** 優先研究「中小型補習班導師」，理由是大型已被系統解決、行政老師是次級訴求。

### Step 7. 真人訪談規劃

**優先訪談對象：**
1. 王老師（台中北屯，已聯絡）
2. 陳老師（高雄三民，已聯絡）

**3 訪談題：**
1. 你最近一次寫家長回報是什麼時候？花多少時間？
2. 你現在用什麼方法整理週間資料？
3. 有沒有試過任何工具或服務？為什麼放棄？

**訪談禁則確認：** 不推銷解法、不問會付多少。

### Step 8. Pain Quality Score

| 維度 | 分數 | 理由 |
| :--- | :--- | :--- |
| 人群具體度 | 5 | 3 位老師都具體 |
| 發生頻率 | 5 | 每週穩定發生 |
| 痛苦強度 | 4 | 已花 5–6 小時 + 情緒成本 |
| 現有解法不滿 | 5 | 3 個 workaround 都被棄用 |
| 證據可信度 | 4 | 已訪談 1 人 + 5 個工具評論 + 3 篇論壇 |
| **總分** | **23 / 25** | （教學鏡子，非決策） |

### Step 9. 真/假判斷

**判斷：** ✅ 真痛點（待訪談 2 人後再升級為 `verified_interview`）

**理由（書面 100 字）：** 三個物理量物件都成立——具體人 3 位、workaround 有名字（LINE+Excel）、痛苦成本明確（5–6 小時 + 情緒）。AI 證據與訪談都指向同一矛盾「個人化 vs 規模化」。最大不確定是是否大型補習班已被系統解決——下一步用王、陳老師訪談排除。

**S2 壓力測試：**
- 可證偽性：王、陳兩位都說「我寧願自己寫」→ 放棄
- 假因排除：「也許老師真痛是怕說壞消息」→ 排除（林老師明說是時間黑洞，不是措辭）
- 定義漂移：Step 1 林老師 vs Step 7 王、陳老師——同類型同情境 ✅

**下一步：** 排王、陳訪談（本週內），訪談完更新狀態到 `verified_interview`，再考慮進階段二 sprint。

**初判 D/R/I（不參與真假判斷）：** Reshape——光加 AI 寫稿助手是補丁，需重設「老師如何在週間記錄」的工序。

### 雙模式輸出

**產出模式：** Validation Status `structured`（待訪談升級到 `verified_interview`）

## 結語

> AI 時代的 0→1，最稀缺的不是工具、不是模型——是**判斷力**。
> 判斷力分兩種：判斷「真假痛點」與判斷「能不能賺錢」。
> 兩者**不可混淆**——混在一起就是 v1 的錯。
> v2 的修正是強制拆階段：階段一只訓練判斷真假，階段二才驗證商業。
> 階段一的篩網不通過 → 不進階段二。
> 階段一通過 → 階段二未必過，但你已經省下 90% 的瞎做時間。
