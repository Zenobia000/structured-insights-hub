## 0. TRIZ 是什麼？（30 秒版）

TRIZ（俄文：Теория решения изобретательских задач，「發明問題解決理論」）由蘇聯工程師 Genrich Altshuller 在 1946 年提出，研究數十萬份專利後，歸納出創新的本質常常是「**解決一組看似不能同時要的兩件事**」。

PainMap Worksheet 不教完整 TRIZ；只借用「**矛盾識別**」這一步作為痛點拆解的鏡子。原因：

- 真痛點背後幾乎都有 trade-off（trade-off 越尖銳，痛點越真）
- 假痛點會「兩件事都想要但其實沒在投入時間 / 金錢去解」
- 矛盾識別逼使用者把「我覺得很煩」拆成「我想要 A，但只能拿 B」的具體取捨

worksheet 把 TRIZ 經典 39 矛盾 + 40 解法縮減成「6 種最常在現代知識工作 / 服務業出現的矛盾」，給初學者用。

## 1. 為何「只能選 1」？

> worksheet 卡 5 過關條件原文（第 290 行）：「**只選 1 個**（複選代表你還沒拆乾淨，退回卡片 3）」。

設計理由：

| 理由 | 說明 |
| :--- | :--- |
| 1. 複選 = 拆得不夠細 | 如果你覺得「想快又想好」+「想客製化又想規模化」都對，代表卡關公式（卡 3）還含有兩層問題沒拆開。回卡 3 拆細，每層分別跑卡 5。 |
| 2. 真痛點通常只有 1 個主矛盾 | 經驗法則：當人「兩件事都想要但只能選 1」時，他每次只會在那一個矛盾上糾結。不是同時在 6 個矛盾上糾結。 |
| 3. 訓練「下決定」的判斷力 | worksheet 整體是判斷力訓練器。若允許複選，使用者會養成「我都選好了」的習慣，喪失逼自己看清核心矛盾的訓練。 |
| 4. 對應 PainCard `triz_id: 1 \| 2 \| 3 \| 4 \| 5 \| 6` | schema 強制單一 `triz_id`，不接受陣列。 |

UI 實作：

- 6 個矛盾以 radio button 呈現（HTML `<input type="radio" name="triz_id">`）
- 不可為 checkbox
- AI 提案 prompt 也內建「不要挑超過 1 個」（`ai_prompt_library.md §3.2` 末段）
- 若 AI 真的挑了 2 個 → 觸發 fallback「請建議我退回卡 3」

### 矛盾 #1 ｜ 想快但又想做得好（Speed vs Quality）

| 屬性 | 內容 |
| :--- | :--- |
| 中文標籤 | 想快但又想做得好 |
| 英文標籤 | Speed vs Quality |
| triz_id | 1 |
| TrizLabel (Schema) | `speed_vs_quality` |

**A 端模板**：「我希望這件事能在 [短時間內] 做完」
**B 端模板**：「但我也希望這件事的成品 [達到某個品質標準]」
**犧牲面通常是**：A（趕時間 → 品質下降）或 B（追求品質 → 時間爆掉）

**產業 / 場景案例（3-5 個）**：

| 場景 | A 端（速度） | B 端（品質） | 通常犧牲 |
| :--- | :--- | :--- | :--- |
| 自由工作者接案 | 「客戶說 3 天要交」 | 「我習慣每件案子做到自己滿意」 | A（熬夜趕但仍然品質掉） |
| 餐廳廚房午餐尖峰 | 「30 分鐘要出 50 份」 | 「擺盤跟調味要對得起本店招牌」 | A（出單但被客人投訴） |
| 醫師看診 | 「每位病人 5 分鐘」 | 「想跟病人解釋清楚再開藥」 | A（解釋變敷衍） |
| 工程師寫測試 | 「明天要交 PR」 | 「測試覆蓋率要 80%」 | B（先放掉測試先 ship） |
| 設計師接 logo | 「客戶 1 週要看初稿」 | 「想多試幾個視覺方向」 | A（只能提 1 個方向） |

**常見誤判**：

| 誤判情境 | 為何誤判 | 應該選 |
| :--- | :--- | :--- |
| 把「想客製化但又想規模化」誤選成 #1 | 客製化的本質是「為每個客戶調整」，不是品質好壞，是 1:1 vs 1:N 的差異 | #2 |
| 把「想專業又想新手懂」誤選成 #1 | 那不是速度問題，是抽象 vs 具象的差異 | #4 |

### 矛盾 #2 ｜ 想客製化但又想規模化（Personalization vs Scale）

| 屬性 | 內容 |
| :--- | :--- |
| 中文標籤 | 想客製化但又想規模化 |
| 英文標籤 | Personalization vs Scale |
| triz_id | 2 |
| TrizLabel (Schema) | `personalization_vs_scale` |

**A 端模板**：「我希望每個 [對象] 都覺得我有針對他做調整」
**B 端模板**：「但我又希望能 [在有限時間 / 資源內服務 N 個對象]」
**犧牲面通常是**：A（用範本 / 罐頭文字 → 對方覺得敷衍）

**產業 / 場景案例**：

| 場景 | A 端（客製化） | B 端（規模化） | 通常犧牲 |
| :--- | :--- | :--- | :--- |
| **補習班老師寫家長 LINE** | 「家長要看見我的孩子被個別關照」 | 「老師一週 2-3 小時寫 30 則」 | A（罐頭被識破） |
| 心理諮商師寫個案紀錄 | 「每個個案的脈絡都不同」 | 「一週 30 個個案要寫完」 | A（紀錄變模板） |
| 業務跟進客戶 LINE | 「客戶要感覺到被重視」 | 「業務一人對 200 個客戶」 | A（群發訊息） |
| 部落客回讀者私訊 | 「想針對每位讀者問題具體回」 | 「一天幾百則訊息」 | A（複製貼上回） |
| 補教家教派功課 | 「每個學生強弱不同」 | 「老師時間有限沒法一個個出題」 | A（一張考卷打天下） |

**worksheet 既有範例對照**：

> **「補習班老師 = #2 個人化 vs 規模化」**（worksheet 第 282-286 行）
> - A 端：家長要看見「我的孩子」被個別關照（具體事件、語氣有溫度）
> - B 端：老師一週只有 2-3 小時可寫 30 則（每則 < 6 分鐘）
> - 犧牲：A 端（罐頭訊息、家長一看就知道沒在用心）

**常見誤判**：

| 誤判情境 | 為何誤判 | 應該選 |
| :--- | :--- | :--- |
| 把「文字要漂亮又要快」誤選成 #2 | 那是 #1 速度 vs 品質 | #1 |
| 把「想自動化又怕失控」誤選成 #2 | 雖然規模化常用自動化，但 #5 是不同層次：你怕失控不等於想客製化 | #5 |

### 矛盾 #3 ｜ 想快但又想正確（Speed vs Accuracy）

| 屬性 | 內容 |
| :--- | :--- |
| 中文標籤 | 想快但又想正確 |
| 英文標籤 | Speed vs Accuracy |
| triz_id | 3 |
| TrizLabel (Schema) | `speed_vs_accuracy` |

**A 端模板**：「我希望這個資訊 / 答案能立刻給出來」
**B 端模板**：「但我也希望這個答案是 [可被驗證 / 不出錯]」
**犧牲面通常是**：B（先給快但錯了還要補救）或 A（要正確但對方等不及）

**產業 / 場景案例**：

| 場景 | A 端（快） | B 端（正確） | 通常犧牲 |
| :--- | :--- | :--- | :--- |
| 客服回應顧客退貨 | 「客戶要當下答覆」 | 「答應退貨前要確認原因」 | A（先答應再說） |
| 會計即時報表 | 「老闆現在要看數字」 | 「數字要對得起報稅」 | A 或 B 都常見 |
| 醫療急診初步診斷 | 「5 分鐘要做決定」 | 「不能誤診」 | 兩端都不能犧牲 → 真痛點 |
| 記者突發新聞 | 「越快發越多流量」 | 「事實要查證」 | B（發了再修正） |
| 法律意見即席給 | 「客戶當場問」 | 「法條要查」 | B（先口頭說大概） |

**#1 與 #3 的差異**：

- #1 是「品質高低」（做得好不好、漂亮不漂亮）
- #3 是「對錯」（事實 / 計算 / 判斷的正確性）
- 若是「美感、體驗、情感品質」→ #1
- 若是「資料正確、判斷正確、不出錯」→ #3

### 矛盾 #4 ｜ 想很專業但又想新手好上手（Expert vs Novice）

| 屬性 | 內容 |
| :--- | :--- |
| 中文標籤 | 想很專業但又想新手好上手 |
| 英文標籤 | Expert vs Novice |
| triz_id | 4 |
| TrizLabel (Schema) | `expert_vs_novice` |

**A 端模板**：「我想要 [專業使用者] 用得深入、有所有進階功能」
**B 端模板**：「但我也想要 [新手 / 第一次來的人] 不被嚇跑」
**犧牲面通常是**：A（簡化介面 → 高手嫌少功能）或 B（功能爆滿 → 新手放棄）

**產業 / 場景案例**：

| 場景 | A 端（專業） | B 端（新手） | 通常犧牲 |
| :--- | :--- | :--- | :--- |
| 健身房教練排課 | 「資深會員要進階課表」 | 「新會員要簡單入門」 | 通常分課線 → 痛點較弱 |
| 寫教學文 / 部落格 | 「想寫得有深度」 | 「想讓初學者看懂」 | A（簡化後資深讀者覺得淺） |
| 開發者工具（CLI vs GUI） | 「進階人要 CLI 全功能」 | 「新手要點按鈕」 | 二選一通常做兩個版本 |
| 醫師說明病情 | 「同行要看技術細節」 | 「家屬要聽得懂」 | B（家屬聽完還是茫然） |
| 銀行 App 操作 | 「老用戶要快速轉帳」 | 「新用戶要逐步引導」 | 常被犧牲：新用戶 |

**常見誤判**：

| 誤判情境 | 為何誤判 | 應該選 |
| :--- | :--- | :--- |
| 把「資料夠多但介面要簡單」誤選成 #4 | 那是「資訊密度」問題，不是專家 vs 新手 | 可能 #1 或 #5 |
| 把「想客製化但又要新手會用」誤選成 #4 | 客製化主軸是 1:1 vs 1:N，不是技能差異 | #2 |

### 矛盾 #5 ｜ 想自動化但又怕失控（Automation vs Control）

| 屬性 | 內容 |
| :--- | :--- |
| 中文標籤 | 想自動化但又怕失控 |
| 英文標籤 | Automation vs Control |
| triz_id | 5 |
| TrizLabel (Schema) | `automation_vs_control` |

**A 端模板**：「我想要 [流程 / 任務] 自己跑、不用我盯」
**B 端模板**：「但我又怕 [出錯 / 越權 / 不符合我的判斷] 我不知道」
**犧牲面通常是**：B（用了自動化但要時時人工檢查）或 A（不敢用自動化還是手動）

**產業 / 場景案例**：

| 場景 | A 端（自動化） | B 端（控制 / 人工判斷） | 通常犧牲 |
| :--- | :--- | :--- | :--- |
| 主管用 AI 寫績效評語 | 「不想花週五一整天寫」 | 「怕 AI 寫得跟個案實況不符」 | B（還是要逐字改） |
| 電商自動回客服訊息 | 「想 24h 回」 | 「怕回錯被客訴」 | 通常 hybrid |
| 醫療 AI 看影像 | 「想加快放射科判讀」 | 「怕 AI 漏看真陽性」 | 通常 AI 輔助 + 醫師確認 |
| 投資人用程式跑單 | 「想 24h 抓機會」 | 「怕程式 bug 把帳戶燒光」 | 設停損限額 |
| 父母用 AI 陪小孩學習 | 「想自己有空閒」 | 「怕 AI 教歪 / 不適齡」 | B（還是要陪在旁邊） |

**常見誤判**：

| 誤判情境 | 為何誤判 | 應該選 |
| :--- | :--- | :--- |
| 把「想用 AI 又怕 AI 寫得罐頭」誤選成 #5 | 那是 #2（自動化是手段，本質是 1:1 vs 1:N） | #2 |
| 把「想自動化又怕學新工具」誤選成 #5 | 那不是失控，是學習成本 | 可能 #4 |

### 矛盾 #6 ｜ 想多嘗試但又怕出包（Experimentation vs Risk）

| 屬性 | 內容 |
| :--- | :--- |
| 中文標籤 | 想多嘗試但又怕出包 |
| 英文標籤 | Experimentation vs Risk |
| triz_id | 6 |
| TrizLabel (Schema) | `experimentation_vs_risk` |

**A 端模板**：「我想要試 N 種 [做法 / 工具 / 方向]」
**B 端模板**：「但我又怕 [試錯成本太高 / 影響現有客戶 / 砸了招牌]」
**犧牲面通常是**：A（不敢試 → 卡在原地）或 B（亂試 → 出包）

**產業 / 場景案例**：

| 場景 | A 端（多嘗試） | B 端（怕出包） | 通常犧牲 |
| :--- | :--- | :--- | :--- |
| 餐廳改菜單 | 「想推 5 道新菜試水溫」 | 「怕老客人不喜歡跑掉」 | A（一年只敢改 1 道） |
| 設計師改 logo | 「想試風格大改版」 | 「怕現有客戶說不像我們」 | A 或 B（看老闆膽量） |
| 廣告主投社群廣告 | 「想 A/B test 10 種素材」 | 「怕預算燒光」 | A（保守只投 2 種） |
| 工程師重構程式 | 「想換掉舊框架」 | 「怕線上出 bug」 | A（卡在 legacy） |
| 老師換教法 | 「想試翻轉教學」 | 「怕學期成績下降被家長罵」 | A（還是用舊方法） |

**常見誤判**：

| 誤判情境 | 為何誤判 | 應該選 |
| :--- | :--- | :--- |
| 把「想自動化但怕失控」誤選成 #6 | 失控是「跑了但結果不對」；出包是「試錯導致負面後果」 | #5 |
| 把「想客製化但怕花太多時間」誤選成 #6 | 那是 #2 個人化 vs 規模化 | #2 |

## 3. 矛盾 vs PainCard 欄位對照

| Schema 欄位 | 取值 | 說明 |
| :--- | :--- | :--- |
| `contradiction.triz_id` | 1-6 | 必填，單選 |
| `contradiction.triz_label` | 上表的中文標籤 | 自動依 `triz_id` 帶入（前端常數） |
| `contradiction.side_a` | 自由文字 | 對應該矛盾的 A 端模板填入 |
| `contradiction.side_b` | 自由文字 | 對應該矛盾的 B 端模板填入 |
| `contradiction.sacrificed` | 'a' / 'b' | 通常會犧牲哪一邊 |

### 範例：林老師案例 → JSON

```json
{
  "contradiction": {
    "triz_id": 2,
    "triz_label": "想客製化但又想規模化",
    "side_a": "家長要看見「我的孩子」被個別關照（具體事件、語氣有溫度）",
    "side_b": "老師一週只有 2-3 小時可寫 30 則（每則 < 6 分鐘）",
    "sacrificed": "a"
  }
}
```

## 5. 卡 5 教學頁面建議呈現順序

```
1. 30 秒短文：為什麼真痛點背後常有矛盾？（引用 Altshuller）
2. 6 個矛盾卡片 + radio button 單選
3. 點開任一矛盾 → 顯示 A 端 / B 端模板 + 3 個產業案例
4. 「請 AI 提案」按鈕（觸發 §3.2 prompt）
5. AI 提案後：使用者可改選或編輯 A 端 / B 端
6. 過關按鈕：當 triz_id 已選 + side_a / side_b 各 ≥ 8 字 + sacrificed 已選
```

## 6. 快速判斷指南（給使用者）

當你不確定該選哪個時，問自己這 3 題：

1. **這個卡關的核心衝突是「資源 / 時間」嗎？**
   - 是 → 候選：#1（速度 vs 品質）、#3（速度 vs 正確）
2. **是「對誰服務」的問題嗎？**
   - 1 對 1 vs 1 對多 → #2（個人化 vs 規模化）
   - 高手 vs 新手 → #4（專家 vs 新手）
3. **是「敢不敢動」的問題嗎？**
   - 怕 AI / 系統失控 → #5（自動化 vs 控制）
   - 怕嘗試新做法出包 → #6（嘗試 vs 風險）

如果 6 題都不像，就是「卡 3 還沒拆乾淨」，回去拆。
